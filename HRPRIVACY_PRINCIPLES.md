# 사이냅 인사평가 시스템 — 개인정보 보호 원칙 (Privacy Principles)

> 이 문서는 본 인사평가 시스템의 핵심 약속을 시스템 설계 단계에서 못박는다.
> 모든 PROMPT 작업은 이 원칙을 위반하지 않는지 점검 후 진행한다.
> 외부 감사·노동청 조사·고객사 실사 대응 시 본 문서를 제출 가능한 수준으로 작성한다.

---

## 원칙 1 — 평가 데이터는 광고·AI 학습에 사용하지 않는다

본 시스템에 저장되는 모든 평가 데이터(이름, 부서, 직책, 평가 점수, 평가 의견, 등급, 첨부 파일 등)는 평가 본연의 목적 외에 사용하지 않는다.

- 평가 데이터를 광고 타겟팅에 활용하지 않는다
- 평가 데이터를 외부 AI 모델 학습 데이터로 사용하지 않는다
- AI 성과 요약 기능에서 사내 LLM에 전송되는 평가 데이터는 응답 생성 즉시 폐기되며, LLM 제공자(현재 사내 Synap LLM)의 학습 데이터로 사용하지 않는다 (사내 LLM 운영 정책으로 보장)
- 외부 광고 SDK·트래커를 본 시스템에 삽입하지 않는다

**구현 의무**:
- 처리방침·표준약관에 명문화 (INFRA-4)
- 외부 SDK 도입 시 이 원칙 위반 여부 검토 필수
- 외부 LLM 사용 검토 시 데이터 사용 정책 확인 필수 (현재는 사내 LLM 사용)

---

## 원칙 2 — 사용자 단위 데이터 분리

본 시스템에 저장되는 데이터는 **사용자(직원)** 단위 및 **회사** 단위로 격리된다.

- 사용자의 평가 데이터는 본인, 본인의 평가자(상사·2차 평가자), 인사팀 외에 접근 불가
- 회사(향후 멀티테넌시 도입 시) 단위로 데이터 완전 격리, 다른 회사의 평가 데이터에 접근 불가
- DB 레벨 격리 보장 (외래키 + 권한 검증 + 회사 단위 필터링)
- 클라이언트 코드는 권한 검증을 신뢰하지 않는다 (서버 측 권한 검증 필수, defense in depth)
- 운영자(서비스 관리자)도 평가 데이터 본체를 직무상 필요한 경우 외에는 열람하지 않는 것이 권장 운영 원칙

**구현 의무**:
- 모든 라우터에서 `req.user`의 권한 확인 후 데이터 반환
- 향후 멀티테넌시 도입 시 모든 테이블에 `tenant_id` 추가, 모든 쿼리에 자동 필터링 적용
- Repository Pattern으로 권한 체크 일원화 (현재 진행 중, INFRA-2A)

---

## 원칙 3 — 떠날 자유 (Right to Leave)

사용자(직원)는 퇴사 후 자신의 평가 데이터에 대한 권리를 행사할 수 있다.

### 데이터 보존 기간
- 재직 중: 평가 데이터 전체 보존
- **퇴사 후 5년 보존 → 자동 삭제** (한국 근로기준법 제42조 3년 보존 의무 + 2년 안전 여유 + 분쟁 대응)
- 보존 기간 종료 시 cryptographically irreversible 영구 삭제 (DB row 삭제 + Object Storage 첨부 파일 삭제)

### 사용자 권리
- 본인의 평가 데이터 전체를 ZIP 또는 PDF로 다운로드 가능 (재직 중)
- 퇴사 시점에 인사팀에 신청하여 평가 데이터 사본을 받을 수 있음
- 보존 기간 내 본인의 평가 데이터 열람 요청 시 인사팀이 응답 (법정 의무)

**구현 의무**:
- 데이터 내보내기 기능 (개인용, 본인 데이터만, 향후 추가)
- 보존 기간 자동 삭제 스케줄러 (`퇴사일 + 5년` 시점 자동 실행)
- 영구 삭제는 DB row 삭제 + Object Storage 객체 진짜 삭제 (단순 `is_active=0` 아닌 실삭제)

**처리방침 명문화 의무**:
- 퇴사 후 보존 기간 5년 명시
- 보존 기간 산정 기준일 명시 (퇴사일)
- 보존 기간 내 사용자 권리(열람, 정정, 사본 발급) 명시

## 원칙 4 — 운영자 접근 정책 (직무 권한 모델)

본 시스템은 인사평가 시스템의 특수성을 고려하여 직무 권한 기반 접근 정책을 적용한다.

### 권한 분리
- **master**: 최고 관리자 (대표이사, 인사팀장 일부) — 모든 시스템 설정 및 평가 데이터 접근 가능
- **admin**: 관리자 (인사팀 일반) — 평가 데이터 접근 및 운영 가능 (시스템 설정 제외)
- **user**: 일반 직원 — 본인 평가 데이터 + 권한 있는 평가 대상자 데이터만 접근

### 평가 데이터 열람 권한 (역할별)

| 시나리오 | 권한 | 정책 옵션 |
|---------|------|----------|
| 본인의 모든 평가 (자평/상사평/2차평) 열람 | ✅ | 평가 기간 한정 열람 옵션 가능 |
| 상사가 부하의 모든 평가 (자평/자기 상사평/2차평) 열람 | ✅ | 평가 기간 한정 열람 옵션 가능 |
| 1차 평가자가 자신이 부하를 평가한 내용 열람 | ✅ | 평가 기간 한정 열람 옵션 가능 |
| 부하가 상사의 평가 의견 (자기와 무관한, 상사 자신의 평가) 열람 | **❌ 어떤 경우에도 불가** | 정책 옵션 없음, 시스템 강제 차단 |
| 인사팀(master/admin)이 모든 평가 데이터 열람 | ✅ | 직무 권한 항상 부여 |

### 평가 기간 비활성화 시 정책
관리자가 특정 평가 기간을 비활성화하면, 해당 기간의 평가 결과는 다음과 같이 처리:
- 일반 직원·상사·2차 평가자: 열람 불가
- 인사팀(master/admin, 단계 무관): 열람 가능

### audit_log 정책
- **변경 audit_log 유지** (INSERT/UPDATE/DELETE)
  - 로그인/로그아웃
  - 평가 제출/승인/반려
  - 평가 데이터 수정 (특히 force-phase 강제 변경)
  - 권한 부여/회수
  - 데이터 삭제
  - 평가 기간 활성/비활성화
  - 사용자 비활성화/활성화
- **열람 audit_log는 미수집** (관리자가 직무상 일상적으로 열람하므로 로그 의미 낮음 + 로그 비대화 방지)
- 보존 기간: 한국 개인정보보호법 시행령 제30조 기준 **최소 1년**, 본 시스템은 **3년 보존** (분쟁 대응 여유)

### 관리자 권한 부여 절차
- **master 권한 부여**: 대표이사 결재 (서면 또는 시스템 내 결재)
- **admin 권한 부여**: 인사팀장 결재
- 권한 부여·회수 사실은 audit_log에 자동 기록

**구현 의무**:
- 변경 audit_log는 이미 구현 완료 (`audit_logs` 테이블, server/index.js)
- 평가 기간 한정 열람 정책 옵션 — 현재 일부 구현, 누락 부분 향후 PROMPT로 보완
- 평가 기간 비활성화 시 부하·상사 열람 차단 — 향후 PROMPT로 점검 및 보완
- 부하의 상사 평가 의견 열람 차단 — 시스템 강제 (라우터 권한 체크 필수)

**처리방침 명문화 의무**:
- "본 시스템의 인사 관련 정보에 접근 가능한 관리자(master, admin)는 인사팀장 등 직무상 권한이 있는 직원에 한정되며, 권한 부여·회수는 대표이사(master) 또는 인사팀장(admin) 결재로 진행된다"
- "본 시스템은 평가 데이터의 변경 이력을 audit_log로 기록하며, 보존 기간은 최소 3년이다"

## 원칙 5 — 사고에 대한 투명성

- 데이터 유출 사고 발생 시 영향받은 사용자(직원)에게 **24시간 이내 알림** + 한국 개인정보보호위원회에 **72시간 이내 신고** (개인정보보호법 의무)
- 정기적 보안 점검 (분기별 1회 이상)
- 보안 사고 발생 시 사후 보고서 작성, 처리방침에 사고 사실 기록

**구현 의무**:
- 사고 발생 시 알림 절차 정의 (이메일 + 시스템 공지)
- 분기별 보안 점검 일정 운영 (운영 시작 후)

## 원칙 6 — 국내 처리 (해외 이전 없음)

본 시스템은 한국 인사 데이터의 민감성을 고려하여 **모든 데이터를 국내에서 처리**한다.

### 데이터 처리 위치
- **호스팅**: NCloud (Naver Cloud Platform), 국내 데이터센터 (춘천·평촌·서울·세종)
- **DB**: NCloud Cloud DB for PostgreSQL, 국내
- **Object Storage**: NCloud Object Storage, 국내
- **AI 성과 요약 LLM**: 사내 Synap LLM, 국내 서버
- **이메일 발송 (향후)**: 국내 SMTP 서비스 (예: 네이버 메일 또는 카카오 메일)

### 국외이전 없음
- 현재 본 시스템에서 한국 외 지역으로 개인정보를 이전하는 절차는 없다
- 향후 글로벌 SaaS 전환 등으로 국외이전이 필요해질 경우, **사전에 본 원칙을 개정하고 사용자 명시적 동의를 받는다**
- 외부 LLM(OpenAI, Anthropic 등) 사용은 본 원칙과 충돌하므로 현재 미사용

**구현 의무**:
- 처리방침에 위탁사 명시 (NCloud + 향후 추가 위탁사)
- 외부 SDK·서비스 도입 시 데이터 처리 위치 확인 필수
- 위탁사 변경 또는 추가 시 처리방침 개정 + 재동의 요청

## 점검 체크리스트 (매 PROMPT 시작 전)

각 PROMPT 작업 시작 전 다음 항목을 점검한다.

- [ ] 이 작업이 사용자(직원)의 평가 데이터를 광고·AI 학습 외부로 보내는가? → No 확인
- [ ] 이 작업이 다른 회사 또는 다른 사용자의 데이터에 접근 가능하게 만드는가? → No 확인
- [ ] 이 작업이 부하가 상사의 평가 의견을 열람할 수 있게 만드는가? → No 확인 (절대 불가)
- [ ] 이 작업이 사용자의 "떠날 자유"를 제약하는가? (보존 5년, 자동 삭제) → No 확인
- [ ] 이 작업이 운영자에게 직무 외 접근 권한을 부여하는가? → No 확인
- [ ] 이 작업이 변경 audit_log를 우회하는가? → No 확인
- [ ] 새로 추가하는 외부 SDK·서비스가 위 원칙과 충돌하는가? → No 확인
- [ ] 이 작업이 국외 데이터 이전을 추가하거나 변경하는가? → No 확인 (본 시스템은 국내 처리 원칙)

위 모두 No 면 진행. 하나라도 Yes 이고 적절한 절차(원칙 개정, 사용자 동의 등)가 없으면 사용자(대표이사)와 협의 후 결정.
